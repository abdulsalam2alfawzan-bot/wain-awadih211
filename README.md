# وين أوديها — إصدار المناطق وOTP وتقييم Google Maps

## التشغيل المحلي

من مجلد المشروع:

```bash
python -m http.server 8080
```

ثم افتح:

```text
http://localhost:8080/index.html
```

تحديد الموقع في المتصفح يعمل على `localhost` أو عبر HTTPS. بيانات المشروع الحالية تحفظ في `localStorage` داخل المتصفح والجهاز نفسه.

## رحلة العميل

1. تحديد الموقع تلقائيًا أو اختيار المنطقة والمدينة يدويًا.
2. إدخال الاسم ورقم جوال سعودي.
3. إنشاء رمز OTP من 6 أرقام وعرضه وملئه تلقائيًا للاختبار، ثم التحقق منه.
4. إدخال بيانات السيارة ووصف المشكلة.
5. الإجابة عن صفر إلى ثلاثة أسئلة توضيحية عند الحاجة.
6. مراجعة التوجيه الفني المبدئي ودرجة الاستعجال والتخصص.
7. الموافقة على مشاركة البيانات اللازمة.
8. إنشاء الطلب وإحالته إلى ورش مناسبة.
9. متابعة الطلب، طلب بديل خلال 24 ساعة، ثم التقييم.

## المناطق والمدن

يدعم `assets/automotive-data.js` المناطق الإدارية الثلاث عشرة و44 مدينة رئيسية واردة في النموذج التشغيلي. يستخدم تحديد الموقع إحداثيات الجهاز لاختيار أقرب مدينة مدعومة، ثم يبقى للعميل حق تعديل المنطقة والمدينة يدويًا قبل إنشاء الطلب.

## OTP التجريبي

- يقبل النظام رقمًا سعوديًا بصيغة `05XXXXXXXX`.
- ينشئ رمزًا من 6 أرقام ويربطه بالرقم ووقت انتهاء.
- يظهر الرمز ويكتب تلقائيًا في الحقل لتسهيل الاختبار.
- يمنع الانتقال حتى التحقق.
- تغيير رقم الجوال يلغي الرمز السابق وينشئ رمزًا جديدًا.

هذا التحقق ليس SMS حقيقيًا ولا يثبت ملكية الرقم إنتاجيًا. قبل الإطلاق يجب نقل إنشاء الرمز والتحقق منه إلى Backend وربطه بمزود SMS أو WhatsApp Business رسمي، مع تحديد المحاولات والمهلة ومنع إعادة الاستخدام.

## Google Places API

### بنية التكامل

- تحفظ سجلات الورش `googlePlaceId` وحالة الربط فقط بصورة دائمة.
- يطلب نموذج الانضمام Google Place ID، وتستطيع الإدارة التحقق منه وربطه بالسجل.
- لا يوضع مفتاح Google داخل HTML أو JavaScript العام.
- `netlify/functions/google-place-details.js` يقرأ المفتاح من متغير البيئة `GOOGLE_MAPS_API_KEY`.
- الواجهة تستدعي `/api/google-place-details` من النطاق نفسه.
- يطلب الخادم الحقول اللازمة فقط من Place Details (New): الاسم، التقييم، عدد المراجعات، رابط الخرائط، العنوان، وحالة النشاط.
- يعرض تقييم Google Maps منفصلًا عن تقييم «وين أوديها».
- يستخدم المتصفح ذاكرة قصيرة داخل الصفحة لمنع الطلبات المكررة، ولا يخزن استجابة Google في `localStorage`.
- عند تعذر الخدمة يظهر رابط الخرائط أو حالة بديلة دون تعطيل رحلة العميل.

### إعداد Netlify

1. فعّل Places API (New) والفوترة في Google Cloud.
2. أنشئ مفتاحًا مخصصًا للخادم وقيّده بـPlaces API وبيئة الخادم المناسبة.
3. أضف متغيرات البيئة:

```text
GOOGLE_MAPS_API_KEY=<server key>
GOOGLE_PLACES_ALLOWED_ORIGINS=https://your-domain.example
```

4. انشر المجلد كاملًا على Netlify؛ يربط `netlify.toml` المسار `/api/google-place-details` بالدالة الخادمية.

### GitHub Pages

GitHub Pages لا يشغّل دوال الخادم. يمكن نشر الواجهة عليه، لكن جلب تقييم Google يتطلب Backend منفصلًا أو Proxy آمنًا على نطاق مناسب. لا تضع مفتاح Places Web Service في كود الواجهة.

### بيانات الورش الحالية

سجلات الاختبار القديمة تحتوي حقل Google Place ID لكنها لا تحمل معرفات حقيقية، حتى لا يتم اختلاق ربط مع منشآت فعلية. طلبات الانضمام الجديدة تتطلب المعرف، وتظهر السجلات غير المرتبطة بحالة واضحة إلى أن تعتمد الإدارة الربط الحقيقي.

## متطلبات الإنتاج

قبل الاستخدام التجاري يلزم Backend وقاعدة بيانات مركزية، OTP رسمي، جلسات وصلاحيات خادمية، حماية من إساءة الاستخدام، سجلات تدقيق مركزية، نسخ احتياطية، وربط رسمي مع واتساب. `localStorage` مناسب للعرض المحلي والتحقق الوظيفي فقط، ولا يزامن البيانات بين أجهزة العميل والورش والإدارة.
